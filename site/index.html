What ktew does

ktew automates the entire Kubernetes The Hard Way tutorial. Your machine is the jumpbox. It spins up 3 Lima VMs (1 control plane, 2 workers) — 4 machines total. It generates a CA and 8 TLS certificates, 6 kubeconfigs, an encryption config, downloads all binaries, bootstraps etcd, the control plane, the workers, configures pod networking, and runs a full smoke test.

Then it destroys everything. VMs gone. Work directory gone. The binary deletes itself. You are not left with a Kubernetes cluster. You are left with a time and a TAP report.

Standard Run Main Category

"Hit Congratul8ions!"

  • Your machine as jumpbox + 3 Lima VMs = 4 machines total.
  • Run ./kthw up. Timer starts at Enter, stops at Congratul8ions!
  • 10/10 passed | 0 failed — or it doesn't count.
  • Full cleanup must complete — nothing left but your time.

Setup (not timed)

  • Go 1.22+ installed
  • Lima may be pre-installed (ktew installs it automatically if missing, but that costs time)
  • Lima VM base images may be cached from a previous run
  • Build the binary: go build -o kthw .
  • No pre-downloaded Kubernetes/etcd/containerd binaries — those download during the run

Platform Sub-Categories

Each platform is a genuine performance variable — different virtualization backends, different architectures, different I/O characteristics. These aren't gimmicks; they're real splits.

macOS (Apple Silicon) ARM64 VMs via VZ framework
macOS (Intel) x86_64 VMs via VZ framework
Linux (ARM64) ARM64 VMs via QEMU
Linux (x86_64) x86_64 VMs via QEMU

Glitchless

Same 4-machine layout. Same rules. But nothing is cached.

  • No pre-installed Lima — ktew installs it from scratch
  • No cached VM base images — limactl prune before the run
  • No pre-downloaded binaries
  • Everything downloads fresh. Your internet connection is now part of your time.
  • Full cleanup must complete — nothing left when it's done.

This is the "real experience" category. It measures what a first-time user actually sees.

What the smoke test verifies

Step 10 runs a full verification suite before cleanup:

  • Secrets are encrypted at rest in etcd (AES-CBC)
  • Deployments work (nginx creates and scales)
  • NodePort services are reachable
  • kubectl exec works inside pods
  • All TLS certificates are valid and components authenticate via mTLS

The cluster is real. It's a real Kubernetes cluster with real PKI, real etcd, real pod networking. It just doesn't exist anymore by the time you read this.

Rules

Timing

  1. Timer starts when you press Enter on ./kthw up
  2. Timer stops when Congratul8ions! prints with 10/10 passed | 0 failed and your time
  3. The run must complete fully — VMs destroyed, work directory cleaned, binary self-deleted. You don't get a cluster. You get a time.
  4. Build time (go build) is not included — only the run itself is timed

Standard Run setup

  1. Go 1.22+ pre-installed
  2. Lima may be pre-installed
  3. Lima VM base images may be cached (default behavior after a previous run)
  4. No pre-downloaded Kubernetes, etcd, containerd, or CNI binaries — those download during the timed run

Glitchless setup

  1. Go 1.22+ pre-installed (only exception)
  2. Lima must NOT be pre-installed — ktew installs it during the run
  3. No cached VM images — run limactl prune if Lima was previously installed, or start from a clean system
  4. No pre-downloaded binaries of any kind
  5. Internet speed is part of your time

Proof & disputes

  1. Proof must include full terminal output showing all 10 TAP steps and the Congratul8ions! finish
  2. Terminal recordings (asciinema, screen capture) or full paste accepted
  3. Cloud VMs as jumpbox are a separate platform category if demand emerges
  4. All disputes settled by Kit

How to Run

# Fork first at github.com/KitSutliff/ktew, then:
git clone https://github.com/YOUR_USERNAME/ktew.git
cd ktew
go build -o kthw .
./kthw up

Timer starts at Enter. Timer stops at Congratul8ions!

You'll see TAP output for each of the 10 steps, then:
